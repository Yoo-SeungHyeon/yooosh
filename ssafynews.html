문제 1: Docker 컨테이너 간 네트워킹 이슈

상황

Docker Compose로 frontend와 backend 컨테이너를 동시에 올렸을 때, 브라우저에서 backend 컨테이너를 찾지 못하는 문제가 발생했습니다. 브라우저 입장에서는 backend가 localhost:8000에 존재하지만, frontend 컨테이너 입장에서는 같은 네트워크이므로 컨테이너명으로 접근해야 했습니다.

해결 방법

Nginx 프록시를 통한 라우팅 해결:

  • • 기존: 절대 경로 (http://localhost:8000 또는 http://back:8000)
  • • 해결: 상대 경로 (/api)로 변경
  • • Nginx에서 /api 요청을 backend로 라우팅 처리
  • • 브라우저와 컨테이너 간 네트워킹 문제 완전 해결
학습 포인트

Docker 네트워킹과 프록시 서버의 중요성을 깨닫게 되었으며, 마이크로서비스 아키텍처에서의 서비스 디스커버리 개념을 실제로 경험할 수 있었습니다.

📦

문제 2: Frontend Build 환경변수 하드코딩 이슈

상황

Frontend를 배포할 때 최적화와 용량 축소를 위해 build를 통해 dist 폴더를 배포하는 방식을 채택했으나, build 과정에서 환경변수가 하드코딩되어 container 실행 시 동적으로 변경할 수 없는 문제가 발생했습니다.

원인 분석

Vue.js build 과정에서:

  • • 환경변수 (VITE_API_BASE_URL 등)가 빌드 타임에 하드코딩됨
  • • dist 폴더의 JS 파일에 실제 URL이 문자열로 박혀버림
  • • Docker args나 environment 옵션이 적용되지 않음
대안 및 학습

환경변수를 런타임에 활용하려면 Node.js 기반 배포나 환경변수 주입 스크립트 등이 필요함을 알게 되었습니다. Build 최적화와 환경 유연성 사이의 트레이드오프를 경험했습니다.
